# LaunchFlow License Admin Design

## 1. Scope and trust boundary

Phase 1 consists of an author-only CLI plus reusable issuing core:

- `tools/license_admin.py`: argparse command surface.
- `tools/license_admin_core.py`: parsing-independent issuing, verification, duplicate detection, and audit history.
- `licensing/request_token.py`: non-secret client request token codec shared with the public client.
- `licensing/license_schema.py`: `lflic-1` schema, expiry, and app-version compatibility helpers.

The admin CLI/core are not imported by the public LaunchFlow application and must not be bundled into public Release artifacts. The signing private key is supplied only through `--private-key` or `LAUNCHFLOW_SIGNING_KEY`; its contents are never printed, copied to AppData, written to audit history, or stored in public configuration.

## 2. Client request token

Current format:

```text
LFREQ1.<base64url canonical JSON>.<SHA-256 checksum prefix>
```

Payload fields:

```json
{
  "schema": "lfreq-1",
  "product": "launchflow",
  "app_version": "0.1.0-beta",
  "machine_id": "...",
  "request_id": "UUID",
  "created_at": "ISO-8601 UTC"
}
```

Base64URL and the checksum detect formatting/copy damage; neither provides encryption or authentication. The request contains no private key, signature, license entitlement, or secret. Legacy Base64 JSON request codes remain parseable and receive a deterministic transition `request_id` for duplicate detection.

## 3. License schema

`lflic-1` includes `license_id`, `request_id`, `product`, `machine_id`, `customer`, `edition`, `features`, `issued_at`, `expires_at`, `request_app_version`, `min_app_version`, `max_app_version`, and `signature`.

- Client-reported version is recorded as request metadata only.
- The author chooses edition, features, expiry, and allowed version range.
- The client verifies signature, machine binding, expiry, product/schema, and version range.
- Existing legacy license fields and verification remain supported.
- No replacement key is generated by the admin CLI.

## 4. Phase-1 CLI

```text
python tools/license_admin.py inspect --request "LFREQ1..."
python tools/license_admin.py issue --request-stdin --customer "张三" --edition beta --days 90
python tools/license_admin.py verify --license test.lic --public-key test-public.pem
python tools/license_admin.py history
```

`inspect` masks `machine_id`. `issue` refuses an already issued `request_id` unless `--force` is explicit; forced issues are recorded. Audit JSONL stores license/request IDs, masked machine ID, customer, edition, issue time, output file, and forced-duplicate status. It does not store request email bodies or private-key data.

## 5. Developer license command

`issue-dev` 是现有人工签发流程的受限快捷入口，不是客户端后门：

```text
python tools/license_admin.py issue-dev --request "LFREQ1..." --output "%LOCALAPPDATA%\LaunchFlow-Dev\licenses\license.lic" --private-key "<explicit-path>"
```

- 强制 `edition=developer`，默认 1095 天但不允许永久授权。
- 私钥仍只能来自 `--private-key` 或 `LAUNCHFLOW_SIGNING_KEY`。
- `--output` 必须明确指定 `.lic` 文件；已有文件必须用 `--force` 才能覆盖。
- request ID 重复仍默认拒绝；`--force` 行为写入审计。
- license 仍由客户端校验签名、机器、有效期和版本范围。
- 审计记录 action、masked machine ID、license/request ID、edition 和输出路径，不记录私钥路径或内容。

## 6. Future independent admin UI

The future UI should call `license_admin_core` rather than shelling out to the CLI. Planned screens/actions:

- pending request review list;
- paste/import and automatically inspect request tokens;
- customer, edition, features, expiry, and version range controls;
- issue button with duplicate-request warning;
- local masked issue history;
- license verification;
- open output directory;
- copy a reply template.

It must be built separately from `LaunchFlow.exe`, stay out of public Releases, run only on the author's machine, remain offline by default, and never display or automatically expose the private key.

## 7. Mail automation boundary

Recommended later workflow:

```text
incoming mail
→ extract LFREQ1 token
→ validate and place in pending-review queue
→ human approval
→ local signing process
→ create reply draft
→ human send confirmation
```

Required controls:

- duplicate `request_id` detection;
- per-mailbox and per-machine rate limiting;
- immutable audit events;
- quarantine for invalid/damaged requests;
- mandatory human approval;
- idempotent retries and failure status;
- no raw private key in the mail service;
- future revocation/blacklist extension point.

Phase 1 must not implement unattended automatic issuing. Mail integration, if added later, may prepare a queue and reply draft but must not sign or send without author approval.
